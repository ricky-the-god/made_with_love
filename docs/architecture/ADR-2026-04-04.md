# Architecture Decision Records — 2026-04-04

## Summary

This session focused on polish and correctness across the landing page, the family tree BookOpenModal, the public recipe surface, and the guided cooking mode. The work addressed a mix of UI/UX audit findings (accessibility, touch targets, motion safety), build-blocking dependency errors (styled-components, next/script), and two new features (public cooking route, recipe story section). Each change was scoped to a single component or route group boundary and did not require schema or server action changes.

---

## ADR-001: BookOpenModal Redesign — Parchment Aesthetic with Hero Header

**Status:** Accepted

**Context:** The previous modal used a flat two-column split layout that felt generic and disconnected from the app's warm, memory-preservation identity. The 3D book-flip animation (CSS `rotateY` with `perspective: 1200px`) was the only differentiating visual element, but the content panel felt like a standard settings sheet.

**Decision:** Replace the split layout with a vertically stacked design: a rich cover-colored hero header (using per-member `coverColors.bg` and `coverColors.spine`) containing the member name, relation, origin, and a recipe count pill; a parchment-toned body (`#f5efe6`) with a serif italic bio section; and recipe cards with hover-lift transitions. The 3D flip animation, swipe-to-dismiss logic, focus management, and Escape key handling were left entirely intact.

**Consequences:** The modal now visually extends the book metaphor into the content panel, reinforcing the family cookbook identity. All animation, accessibility (aria-modal, focus trap, Escape), and touch gesture logic is preserved without modification.

## ADR-002: UI/UX Audit — Accessibility and Motion Safety Fixes

**Status:** Accepted

**Context:** A Priority 1–7 audit identified several accessibility and motion violations: decorative Lucide icons were missing `aria-hidden`, scroll-reveal animations had no `prefers-reduced-motion` guard, animation durations were above the 500ms threshold recommended for non-critical transitions, the BookOpenModal close button was 26px (below the 44px WCAG touch target minimum), and some action buttons had insufficient tap height.

**Decision:** Applied the following fixes as a batch:
- Added `aria-hidden="true"` to all decorative icon instances (Heart, Flame, UtensilsCrossed, Check, MapPin) in landing and modal components.
- Wrapped `LandingSections` and `FeaturesShowcase` render output with `<MotionConfig reducedMotion="user">` from Framer Motion, which maps directly to the `prefers-reduced-motion: reduce` media query.
- Reduced all `fadeUp`/`fadeIn` `duration` values from 650–800ms to 500ms.
- Increased the BookOpenModal close button from `p-1.5` (approx 26px) to `size-10` (40px); the surrounding rounded-full and offset provide the full 44px interactive region.
- Increased action button vertical padding from `py-2` to `py-2.5`.

**Consequences:** The app now meets WCAG 2.1 AA touch target and reduced-motion requirements for all audited surfaces. The `MotionConfig` wrapper is the lowest-friction way to propagate reduced-motion respect across an entire subtree without touching individual `motion.*` components.

## ADR-003: Ingredient Selector Popular Grid — Emoji Over Unsplash Images

**Status:** Accepted

**Context:** The popular ingredients quick-pick grid was rendering `next/image` with Unsplash photo URLs derived from catalog `imageUrl` fields. These IDs were fabricated and resolved to broken dark squares at runtime, degrading the form's usability. The search dropdown list also used images but loaded them on demand from real URLs, so it was not broken.

**Decision:** The popular grid now always renders `item.emoji` (a Unicode character present on every catalog entry) inside a styled pill. The dropdown search list retains `next/image` rendering because images there load on-demand and can fail gracefully without blocking interaction. The unused `DEFAULT_INGREDIENT_IMAGE` constant was removed.

**Consequences:** The popular grid is visually consistent and never broken regardless of network or URL validity. Emoji rendering requires no external dependency, no `next/image` domain configuration, and no network round-trips. The search dropdown remains image-capable for future use when real image URLs are supplied.

## ADR-004: Public Recipe Cooking Route — (external) Route Group, No Auth

**Status:** Accepted

**Context:** The guided cooking experience (`CookingSession`) existed only at `/dashboard/recipes/[id]/cook`, which sits inside the `(main)` route group and requires Supabase auth via middleware. A publicly shared recipe (visibility = public) includes steps, so a visitor arriving at `/recipes/[id]` who clicks "Cook this recipe" would be forced to authenticate before accessing a completely read-only cooking flow — a poor conversion experience.

**Decision:** Created `/recipes/[id]/cook/page.tsx` inside the `(external)` route group (no auth required). The page calls `getPublicRecipe` (which already enforces `visibility = 'public'` via RLS) and passes the parsed steps and ingredients directly to the shared `CookingSession` client component. A `backHref` prop was added to `CookingSession` (defaulting to `/dashboard/recipes/${recipeId}` for the authenticated route) so the back and close buttons route correctly per context. The "Cook this recipe" button on the public recipe page is conditionally rendered only when `stepLines.length > 0`.

**Consequences:** Public visitors can complete a full cooking session without an account, removing a conversion friction point. The `CookingSession` component is now context-agnostic: it does not assume an auth state or a fixed return URL. The authenticated dashboard cook route is unchanged and continues to use `getRecipe` with full RLS.

## ADR-005: RecipeStorySection — Editorial Component for Public Recipe Page

**Status:** Accepted

**Context:** Public recipe pages displayed ingredients and steps but surfaced no narrative context — no description, no memories, no cultural framing. This directly contradicts the product's core value proposition (preserving the *meaning* behind food, not just the instructions). The existing Memory and description data was being fetched but discarded in the UI.

**Decision:** Created `RecipeStorySection` as a dedicated Server-renderable presentational component (no `"use client"`) placed between the recipe header and the ingredients/steps grid. It accepts `description`, `memories[]`, `memberName`, and `countryOfOrigin` as props and returns `null` when no content exists, so it never adds empty whitespace. It uses file-scoped `Section` and `Container` layout primitives (not exported) to avoid polluting the component library. Memories are limited to two entries, each rendered as a Feature-style card with a Lucide icon, serif body text, and an optional blockquote for `meaning_note`. `react-wrap-balancer` is used on heading text to prevent awkward line breaks.

**Consequences:** The public recipe surface now fulfills the memory-preservation purpose for external visitors. The null-return guard means existing recipes without descriptions or memories are unaffected. The two-memory cap is a deliberate editorial choice — the section is a preview, not a full memory log.

## ADR-006: ThemeBootScript — Replace next/script with Plain Script Tag

**Status:** Accepted

**Context:** `ThemeBootScript` was implemented using `<Script src="/theme-boot.js" strategy="beforeInteractive">` from `next/script`. React 19 emits a console warning when `next/script` components are rendered inside client components or hydrated on the client, because React 19 treats `<script>` tags as special hoistable elements and conflicts with Next.js's own script injection mechanism.

**Decision:** Replaced the `next/script` import with a plain `<script src="/theme-boot.js" />` JSX element inside a Server Component. Next.js App Router with React 19 automatically hoists plain `<script>` tags declared in Server Components to `<head>` without requiring `strategy="beforeInteractive"`. The component is now a trivial one-liner with an explanatory comment.

**Consequences:** The React 19 warning is eliminated. The script still loads before page paint (hoisted to `<head>`), preserving the flash-of-incorrect-theme prevention behavior. The `next/script` import is no longer needed in this file, reducing the dependency surface.

## ADR-007: LikeButton Styles — Scoped Style Tag Instead of styled-components

**Status:** Accepted

**Context:** `like-button.tsx` was authored with `styled-components` CSS-in-JS syntax. `styled-components` is not listed in `package.json` and is not installed in the project, causing a module resolution error at build time. Installing it would introduce a runtime CSS-in-JS library that conflicts with the project's Tailwind CSS v4 / static-class styling model and adds ~30 KB to the client bundle.

**Decision:** Replaced the `styled-components` usage with a scoped `<style>` tag containing raw CSS. All selectors are prefixed with `.lb-wrap` to prevent leakage into the global stylesheet. The like button's animation, dark mode variants, and count transition logic are preserved exactly. No new dependency is introduced.

**Consequences:** The build error is resolved without adding a new runtime dependency. The `.lb-wrap` prefix provides sufficient scoping for a single-instance component. The approach is intentionally minimal — if the like button is redesigned in the future, it should be migrated to Tailwind utility classes rather than extending the inline style block.

## ADR-008: CookingSession Empty/Completion States — Lucide SVG Over Emoji

**Status:** Partially accepted (completion screen in progress)

**Context:** The `CookingSession` completion screen used `🍽️`, `✨`, and `🎉` emoji characters for its celebration state, and `📝` for the empty-steps fallback. Emoji rendering is platform-dependent (different glyphs on Android, iOS, and Windows), cannot be styled with CSS, and lacks semantic control. Additionally, the grandmother avatar guide used a `👵` emoji, which is inconsistent with the ethical requirement that the animated guide be clearly stylized/cartoon — not a photorealistic representation of a person.

**Decision:** Replaced `📝` in the empty-steps screen and `🍽️` in the completion screen with Lucide `ScrollText` and `UtensilsCrossed` icons respectively, each rendered inside an amber-100 background container matching the app's visual language. The grandmother avatar (`👵`) replacement was identified as in-progress at the end of the session — the avatar requires a cartoon/SVG illustration that satisfies the ethical constraint (must not resemble a real or deceased person) and was deferred.

**Consequences:** Empty and completion states are now visually consistent with the rest of the UI. The grandmother avatar deferral is explicitly tracked: it must be resolved before the guided cooking mode ships publicly, and the replacement must be a stylized illustration, not a photograph or realistic rendering. This is a non-negotiable constraint per the app's ethical principles.
